# Stage 1 — mobile-first UX-каркас и handoff-пакет

Документ переводит исходное ТЗ в рабочий IA/UX deliverable-пакет для wireframes, прототипа и следующего UI/dev handoff.

## 1. Рамка этапа и рабочие допущения

### 1.1 Что считается результатом этапа
- согласованный sitemap с приоритетами P0/P1/P2;
- набор золотых сценариев с ясным целевым действием;
- mobile-first логика навигации и сквозных CTA;
- структура обязательных модулей для P0-страниц;
- low-fi mobile wireframes для всех P0-экранов;
- кликабельные end-to-end прототипы ключевых сценариев;
- testing pack для быстрых UX-проверок;
- handoff-пакет для UI и разработки.

### 1.2 Рабочие допущения для текущего проекта
1. Канонический slug страницы контактов для текущей реализации: `/contacts`.
2. В кодовой базе уже существуют дополнительные маршруты `/format` и `/topics/{slug}`. Они не входят в канонический MVP sitemap этого этапа и рассматриваются как supporting/legacy routes до отдельного IA-решения.
3. В кодовой базе уже есть дополнительные сервисные форматы вне P0. Финальная P2-таксономия (`couples` / `teens` из ТЗ или альтернативные форматы из текущего контента) требует отдельного продуктового решения и не блокирует Stage 1.

### 3.4 Канонические правила sitemap
1. Любая P0-страница либо ведёт к `/booking`, либо снижает тревогу перед переходом в `/booking`.
2. Из основных входов пользователь должен попадать в запись максимум за 2 шага.
3. Legal-страницы не изолированы: они встроены в сценарий записи и имеют явный возврат.
4. P1-страницы не должны размывать основной путь. Их роль: поддержать решение, а не увести в глубину.

## 4. Golden user flows

| ID | Вход | Цель пользователя | Маршрут | Риски и вопросы | Целевое действие |
|---|---|---|---|---|---|
| F1 | Search / рекомендация / direct link | Быстро понять, подходит ли специалист, и записаться | `/` или `/services/individual` -> `/booking` -> `/booking/confirmed` | "Подходит ли мне формат?", "Сколько это стоит?", "Насколько сложно записаться?" | Отправка формы записи |
| F2 | Главная | Сначала проверить доверие к специалисту | `/` -> `/about` -> `/about/boundaries` -> `/booking` | "Кто этот человек?", "Какие рамки и границы?", "Безопасно ли писать?" | Переход к записи после снижения тревоги |
| F3 | Статья или FAQ | Уточнить, подходит ли услуга моему запросу | `/blog/{slug}` или `/faq` -> `/services/individual` -> `/booking` | "Работает ли специалист с моим запросом?", "Как проходит первая встреча?" | Выбор времени / отправка формы |
| F4 | Страница записи | Проверить legal и условия, не потеряв решение записаться | `/booking` -> `/privacy` или `/consent` или `/terms` или `/about/boundaries` -> `/booking` | "Что будет с моими данными?", "Какие правила отмены?", "Что именно я подтверждаю?" | Возврат в запись и отправка формы |
| F5 | Главная или контакты | Сначала задать вопрос, потом записаться | `/` или `/contacts` -> `/contacts` -> `/faq` -> `/booking` | "Можно ли написать до записи?", "Что делать, если я пока не уверен(а)?" | Переход из вопроса в запись |
| F6 | Письмо / внешняя ссылка / напоминание | Уточнить перенос или отмену | `/terms` -> `/contacts` или `/booking` | "Как перенести?", "Нужно ли снова записываться?" | Связаться или перейти к новой записи |

### 4.1 Правила проектирования flow
1. В каждой ветке должен существовать один очевидный следующий шаг.
2. Любой "осторожный" сценарий должен возвращать в `/booking` без ощущения потери контекста.
3. Страницы `/privacy`, `/consent`, `/terms` и `/about/boundaries` работают как поддержка решения, а не как тупиковые документы.
4. Нельзя прятать перенос/отмену и условия согласия ниже отправки формы без видимых ссылок рядом с CTA.

## 5. Навигационная модель

### 5.1 Header / mobile menu

Порядок пунктов:
1. `Главная`
2. `О специалисте`
3. `Услуги`
4. `FAQ`
5. `Контакты`
6. `Запись`

### 5.2 Поведение навигации
- На mobile меню открывается полноэкранной или panel-overlay навигацией с тем же порядком пунктов.
- Пункт `Запись` визуально выделен как primary CTA.
- Активный раздел маркируется, но не конкурирует с CTA.
- Дополнительные supporting routes (`/format`, `/topics/{slug}`) не обязаны отображаться в основной навигации Stage 1.

### 5.3 Sticky CTA
- На mobile для контентных страниц P0 и P1 показывается sticky CTA `Записаться`.
- На `/booking` sticky CTA заменяется на локальный submit-контрол внутри формы, чтобы не плодить две конкурирующие кнопки.
- На legal-страницах sticky CTA может быть `Вернуться к записи`, если страница открыта как часть flow F4.

### 5.4 Footer

Обязательный состав footer:
1. `Контакты`
2. `Privacy`
3. `Consent`
4. `Terms`
5. `Cookies` опционально после появления страницы

### 5.5 Сквозные CTA

`Записаться` обязателен на:
- `/`
- `/services`
- `/services/individual`
- `/about`
- `/about/boundaries`
- `/faq`
- mobile menu
- mobile sticky CTA

## 7. Low-fi mobile wireframes

### 7.1 Список обязательных экранов P0
1. `P0-01 Home / default`
2. `P0-02 Services / default`
3. `P0-03 Service Individual / default`
4. `P0-04 About / default`
5. `P0-05 Boundaries / default`
6. `P0-06 Booking / default`
7. `P0-07 Booking / validation error`
8. `P0-08 Booking / legal return`
9. `P0-09 Booking Confirmed / success`
10. `P0-10 Privacy / default`
11. `P0-11 Consent / default`
12. `P0-12 Terms / default`

### 7.2 Wireframe-логика по экранам

#### `/`
1. Первый экран: спокойный hero с одной primary-кнопкой `Записаться` и одной secondary-ссылкой `Как проходит работа`.
2. Ниже без каруселей и скрытых вкладок: trust bar и 6-8 карточек запросов.
3. Перед финальным CTA обязательно показать короткий блок про границы и безопасность.

#### `/services`
1. Сразу показать, что основной маршрут MVP ведёт в `Индивидуальную работу`.
2. Нефинализированные форматы оформить как secondary / soon, не давать им ломать сценарий.
3. Внизу страницы повторить CTA в запись и ссылку на контакты для уточнения.

#### `/services/individual`
1. Above the fold: краткое обещание формата, длительность, стоимость и CTA.
2. Ниже показать "кому подходит / не подходит", чтобы снять риск ошибочного self-selection.
3. Правила переноса и legal должны быть рядом с pricing, а не только в footer.

#### `/about`
1. Первый экран должен снижать тревогу, а не продавать.
2. Квалификация и подход читаются короткими блоками, без длинного сплошного текста.
3. Блок `Границы и безопасность` должен быть виден до финального CTA.

#### `/about/boundaries`
1. В верхней части объяснить конфиденциальность простым языком.
2. Отдельно показать исключения и границы связи вне сессий.
3. Финал экрана: три legal-ссылки и одна заметная кнопка возврата в запись.

#### `/booking`
1. Первый экран: пояснение, что форма короткая и займёт менее минуты.
2. Формат и слот выбираются до контактных данных.
3. Поля формы:
   - имя или псевдоним;
   - email и/или телефон/мессенджер;
   - комментарий опционально.
4. Под чекбоксом согласия обязательно три явные legal-ссылки.
5. Короткие правила переноса/отмены расположены рядом с submit.

#### `/booking/confirmed`
1. На первом экране подтвердить факт отправки и убрать неопределённость "ушло ли сообщение".
2. Ниже дать 2 ясных действия: `Добавить в календарь` и `Связаться`.
3. Ссылка на перенос/отмену должна быть видимой, но вторичной.

#### `/privacy`, `/consent`, `/terms`
1. Первый экран каждой legal-страницы содержит human summary в 3-5 строк.
2. Далее информация разбивается на короткие секции, без юридической стены текста на первом экране.
3. Возврат в `/booking` должен быть доступен как вверху, так и внизу wireframe.

### 7.3 Обязательные состояния и ошибки

#### `Booking / validation error`
- не выбран формат;
- не выбран слот;
- не заполнено поле контакта;
- не отмечен consent;
- ограничение rate limit / техническая ошибка отправки.

#### Правила показа ошибок
1. Ошибка показывается рядом с конкретным полем человеческим языком.
2. Уже введённые данные не сбрасываются.
3. Ошибка не прячет legal-ссылки и не блокирует доступ к `/privacy`, `/consent`, `/terms`.
4. После возврата с legal-страницы wireframe должен визуально напоминать, что пользователь находится внутри сценария записи.

## 8. Prototype flows

| Flow | Что нужно собрать в кликабельном прототипе | Критерий прохождения |
|---|---|---|
| PF1 | `Главная -> Запись -> Подтверждение` | Пользователь доходит до success state без тупиков |
| PF2 | `Главная -> О специалисте -> Границы -> Запись` | Осторожный маршрут не длиннее 3 экранов до booking |
| PF3 | `Статья/FAQ -> Услуга -> Запись` | Контентный вход конвертируется в выбор услуги и запись |
| PF4 | `Запись -> Privacy/Consent/Terms/Boundaries -> Запись` | Legal-проверка не ломает сценарий и возвращает пользователя в booking |

### 8.1 Что должно быть кликабельным
- primary CTA на каждом P0-экране;
- вторичные переходы `О специалисте`, `Границы`, `FAQ`, `Контакты`;
- все legal-ссылки из `/booking`;
- явный возврат из `/privacy`, `/consent`, `/terms`;
- sticky CTA на mobile;
- состояние `submitted` после успешной записи.

## 9. Testing pack

### 9.1 Цель мини-тестов
Проверить, может ли новый пользователь:
1. быстро понять, чем помогает специалист;
2. без лишней тревоги перейти к записи;
3. найти правила конфиденциальности и отмены до отправки формы;
4. завершить сценарий записи без критических барьеров.

### 9.2 Состав выборки
- 5-7 респондентов;
- минимум 3 человека с опытом тревоги перед первым обращением;
- минимум 2 человека без предыдущего опыта психотерапии;
- 1-2 человека, склонных читать условия и legal до отправки формы.

### 9.3 Тестовые задачи
1. "Представьте, что вы впервые ищете психолога. Поймите, подходит ли вам специалист, и попробуйте записаться."
2. "Сначала проверьте, как специалист работает и какие у него профессиональные границы."
3. "Найдите, что будет с вашими данными после отправки формы."
4. "Проверьте правила отмены или переноса встречи."
5. "Если пока не готовы записываться, найдите, как задать вопрос."

### 9.4 Чек-лист наблюдения
- Сколько шагов до записи от основного входа.
- Замечает ли пользователь CTA `Записаться` без подсказки.
- Понимает ли пользователь, чем `/about` отличается от `/about/boundaries`.
- Находит ли пользователь privacy/consent/terms до отправки формы.
- Видит ли пользователь условия переноса рядом с действием.
- Понимает ли пользователь, что запись не является экстренной помощью.
- Замечает ли пользователь альтернативный путь через контакты.

### 9.5 Фиксация проблем

| Severity | Критерий |
|---|---|
| P0 | Блокирует запись, скрывает legal, ломает возврат в booking или вызывает критическую неопределённость |
| P1 | Не блокирует запись, но заметно повышает тревогу, удлиняет сценарий или провоцирует ошибки |
| P2 | Косметическая проблема, не влияющая на completion сценария |

### 9.6 Решение о готовности к handoff
- если найден хотя бы один P0-барьер, wireframes и prototype дорабатываются до передачи в UI;
- P1-проблемы группируются и приоритизируются до следующего цикла;
- handoff допускается только если все 4 обязательных flow проходят end-to-end без P0-проблем.

## 10. Handoff для UI и разработки

### 10.1 Структура Figma-файла
- `00_Sitemap`
- `01_User_Flows`
- `02_Wireframes_Mobile_P0`
- `03_Wireframes_Mobile_P1`
- `04_Modules`
- `05_States_And_Errors`
- `06_Prototype_Flows`
- `07_Testing_Notes`
- `08_Handoff`

### 10.2 Правила именования экранов
- формат: `P{priority}-{nn} {Screen} / {State}`
- примеры:
  - `P0-01 Home / default`
  - `P0-06 Booking / default`
  - `P0-07 Booking / validation error`
  - `P0-09 Booking Confirmed / success`

### 10.3 Правила именования модулей
- формат: `M{nn}_{Module_Name}`
- примеры:
  - `M01_Hero_CTA`
  - `M06_Boundaries_Safety`
  - `M10_Booking_Form_Consent`

### 10.4 Что нужно аннотировать для UI
1. Основной CTA на каждом экране.
2. Sticky CTA и условия его показа/скрытия.
3. Legal-ссылки рядом с формой записи.
4. Поведение экранов ошибок и success state.
5. Возврат в `/booking` из всех legal-страниц.

### 10.5 Что нужно аннотировать для разработки
1. Канонические маршруты: `/`, `/services`, `/services/individual`, `/about`, `/about/boundaries`, `/booking`, `/booking/confirmed`, `/privacy`, `/consent`, `/terms`, `/faq`, `/contacts`.
2. Booking form должна поддерживать минимальный набор полей и отдельный consent checkbox.
3. Legal-ссылки из `/booking` не должны быть скрыты за accordion/modal без явной причины.
4. Ошибки валидации должны быть inline и не сбрасывать введённые данные.
5. Страница подтверждения должна явно закрывать сценарий и предлагать следующий шаг.

### 10.6 Список открытых вопросов
1. Финализировать post-MVP taxonomy для расширенных услуг: `couples` / `teens` из ТЗ или текущие альтернативные форматы из контента.
2. Решить судьбу supporting route `/format`: удалить из главной навигации, архивировать или слить с `/services`.
3. Подтвердить, будет ли MVP использовать реальный календарь/слоты или псевдослоты без внешней интеграции.
4. Провести юридический просмотр страниц `/privacy`, `/consent`, `/terms` перед production.
5. Подтвердить реальный список каналов связи и SLA ответа для `/contacts`.

### 10.7 Список допущений
1. Mobile-first wireframes являются первоисточником для дальнейшего desktop UI.
2. Контент на legal-страницах может быть уточнён юридически без пересборки основной IA.
3. В MVP мультиязычность не проектируется.
4. Если пользователь попал на legal-страницу из записи, приоритет возврата в `/booking` выше любых вторичных CTA.

## 11. Definition of Done для Stage 1

Этап считается закрытым, если:
1. Все P0-страницы описаны в sitemap и существуют как mobile-first wireframes.
2. Все 4 обязательных flow собраны кликабельно end-to-end.
3. Из основных входов пользователь попадает в запись максимум за 2 шага.
4. Privacy / Consent / Terms встроены в сценарий и имеют явный возврат.
5. После мини-тестов не остаётся P0-проблем, блокирующих запись.
