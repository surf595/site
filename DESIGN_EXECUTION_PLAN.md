# Поэтапное применение задач (дизайн + UX + handoff)

Документ переводит `DESIGN_TEAM_BRIEF.md` в конкретный рабочий трек для команды.

## Этап 1 — Direction / Moodboard
**Цель:** зафиксировать визуальное направление без расфокуса.

**Задачи:**
- собрать 2–3 moodboard-концепции;
- выбрать 1 финальное направление;
- утвердить цветовую базу и tone-интонацию интерфейса.

**Артефакты:**
- Figma page: `01_Moodboard`;
- 1-page rationale с решениями и отклонёнными альтернативами.

**DoD:**
- выбран 1 финальный visual direction;
- подтверждены принципы: calm, editorial, trust-first, no aggressive marketing.

---

## Этап 2 — Базовая дизайн-система
**Цель:** создать UI-kit, пригодный для P0/P1 и разработки.

**Задачи:**
- типографическая шкала (H1-H3, body, micro);
- цветовые токены (base/neutral/accent);
- сетка и контейнеры (mobile/tablet/desktop);
- кнопки, формы, карточки, legal-note блоки;
- интерактивные состояния (hover/focus/active/disabled/error/success).

**Артефакты:**
- Figma page: `02_Design_System`;
- таблица token naming для handoff.

**DoD:**
- UI-kit покрывает минимум 90% P0-компонентов без ad-hoc стилей.

---

## Этап 3 — Mobile P0 UI
**Цель:** спроектировать полный mobile-first UI для всех P0 страниц.

**Задачи:**
- собрать mobile макеты `/`, `/services`, `/services/individual`, `/about`, `/about/boundaries`, `/booking`, `/booking/confirmed`, `/privacy`, `/consent`, `/terms`;
- встроить trust/legal блоки в UX, а не выносить «в хвост»;
- обеспечить один главный CTA на экран.

**Артефакты:**
- Figma page: `03_Mobile_P0`.

**DoD:**
- все P0 экраны есть в mobile версии;
- путь до записи с ключевых страниц ≤ 2 шага.

---

## Этап 4 — Desktop адаптация
**Цель:** перенести mobile-логику на desktop без потери приоритетов.

**Задачи:**
- адаптировать P0 экраны под desktop;
- сохранить приоритет CTA и читабельность;
- избежать визуального шума и перегрузки.

**Артефакты:**
- Figma page: `04_Desktop_P0`.

**DoD:**
- desktop не ломает mobile-приоритеты и юридические маршруты.

---

## Этап 5 — Прототип ключевых сценариев
**Цель:** проверить кликабельность критичных путей.

**Сценарии:**
1. Home → Booking → Confirmed
2. Home → About → Booking
3. FAQ/Blog/Topic → Service → Booking
4. Booking → Privacy/Terms/Boundaries → Booking

**Артефакты:**
- Figma page: `05_Prototype_Flows`.

**DoD:**
- все 4 сценария проходят end-to-end;
- нет тупиков и «обрывов» маршрутов.

---

## Этап 6 — Handoff в разработку
**Цель:** передать дизайн без двусмысленности и скрытых допущений.

**Задачи:**
- подготовить спецификации spacing/typography/colors/states;
- зафиксировать поведение sticky CTA, меню, формы и ошибок;
- выделить TODO зоны (контент/юридическая финализация).

**Артефакты:**
- Figma page: `06_Handoff`;
- handoff-note (открытые вопросы, зависимости, риск-зоны).

**DoD:**
- разработка может реализовать интерфейс без повторного проектирования.

---

## Контроль приёмки всего этапа
- Визуально: спокойствие, зрелость, доверие, ясность.
- Mobile-first реализован фактически, а не декларативно.
- P0 маршруты и legal-возвраты читаются и кликаются.
- Форма записи и error/success состояния спроектированы.
- UI-kit масштабируем для P1/P2.
